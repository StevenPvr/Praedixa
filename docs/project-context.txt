## 0) EXECUTIVE SUMMARY (1 page max)

**Praedixa** : Système d’exploitation économique de la capacité: prévoir, arbitrer, prouver

**Ce que Praedixa vend :** une couche “Ops/DAF” au-dessus des outils existants (WFM/ERP/tableurs) qui **détecte à J+3/J+7/J+14 les risques de rupture de couverture**, **explique les drivers actionnables**, et **quantifie l’arbitrage coût vs service** via scénarios, avec **preuve d’impact** (0% / 100% / réel).
**Pourquoi c’est premium :** parce que ce n’est pas un dashboard : c’est un **service contractualisé** (data canonique rejouable + modèles calibrés + moteur économique + protocole d’attribution + proof pack), déployable **sans projet SI lourd** et **audit-ready** (versioning, traçabilité, decision log).
**Orientation go-to-market actuelle (ajout) :** entrée commerciale via un **service cheval de Troie** centré sur le **forecasting staff** (périmètre restreint, time-to-value rapide), puis extension vers le **full package Praedixa** (risque -> drivers -> scénarios coût/service -> couverture -> preuve). **ICP acquisition prioritaire :** multi-franchisés de fast-food moyen (réseaux multi-sites).

**Bénéfices mesurables (cas 10 M€ CA, hypothèses ci-dessous)**

- **Réduction “last-minute”** (HS + intérim urgent + désorganisation) : **-10% à -20%**.
- **Stabilisation service** : baisse des créneaux en sous-couverture et des pénalités/expedite : **-15% à -30%** (sur les coûts liés au non-service).
- **Décisions plus tôt** : bascule d’actions de J-0/J-1 vers J+3/J+7 (moins cher, plus d’options).
- **DAF-ready** : coût complet par option, hypothèses versionnées, preuve mensuelle.
- **Amélioration continue** : erreurs → ajustements (règles/features/coûts) → meilleure précision & adoption.

**Risques**

- Data trop incomplète (charge/couverture/réalisé) → modèle fragile.
- Adoption : si personne ne “consomme” les alertes (rituels + ownership), pas de valeur.
- Contraintes terrain non formalisées (compétences, lead time intérim, caps HS) → scénarios “théoriques”.
- Mauvaise calibration → trop d’alertes → désactivation.
- Gouvernance faible (pas de sponsor Ops+DAF) → pas de décision log → pas de preuve.

**Conditions de succès**

- Exports **agrégés** disponibles (site×créneau), refresh **quotidien**.
- Règles & coûts **validés** (DAF/Ops) et **versionnés**.
- Rituels : revue J+3 (quotidien), J+7 (2×/semaine), J+14 (hebdo).
- Decision log obligatoire (même minimal) + protocole d’attribution.
- Démarrage wedge : 4–6 sites, puis rollout.

**Chiffrage ROI annuel – ordre de grandeur (cas 10 M€)**

- **Baseline “coûts sensibles Praedixa” (hypothèses)** : ~**1,1 M€ / an** (HS + intérim urgent + coûts de non-service + frictions opérationnelles attribuables).
- **Gain crédible** : **0,24 M€ à 0,45 M€ / an** (≈ **2,4% à 4,5% du CA**).
- **Hypothèses clés** : (i) last-minute 0,70–0,90 M€/an, (ii) non-service 0,15–0,30 M€/an, (iii) réduction 10–20% et 15–30% respectivement, (iv) pas de “miracle” sur le volume de travail, seulement sur le mix et le timing.

## 2) MODÈLE OPÉRATIONNEL DU CLIENT (cas logistique 10M€)

### 2.1 Description (profil plausible)

- **Entreprise** : prestataire logistique / entreposage (prépa + expédition + retours), France.
- **Réseau** : **6 sites** (S1…S6) de 4 000 à 12 000 m².
- **Créneaux** : **2 shifts/jour**, **6 jours/semaine**
  - AM : 06:00–14:00
  - PM : 14:00–22:00
- **Compétences (agrégées)** : Réception, Préparation, Expédition, Inventaire/retours.
- **Métriques de charge** (par site×shift) : lignes préparées, colis, palettes, commandes, retours.
- **Métriques de service** : OTIF / expéditions “cut-off” respectées, backlog fin de shift, retards transport, taux d’annulations.

### 2.2 Baseline coûts (hypothèses chiffrées)

**Capacité & coûts internes**

- Effectif : **140 ETP ops**
- Heures/an : 140 × 1 610 = **225 400 h/an**
- Masse salariale chargée : 140 × 32 000 = **4,48 M€ / an**
- Coût horaire chargé interne : 4,48 M€ / 225 400 h = **19,9 €/h**

**Last-minute**

- Heures sup : **12 000 h/an**, coût HS = 12 000 × (19,9 × 1,25) = **0,299 M€**
- Intérim total : **18 000 h/an**
  - coût intérim = 18 000 × 30 € = **0,540 M€**
  - dont “urgent” (40%) : 7 200 h ; surcoût urgence (hyp. +10%) = 7 200 × 30 × 10% = **0,022 M€** (petite ligne, mais signal utile)
- Coûts de désorganisation (hyp. conservatrice) : **0,12–0,20 M€ / an**
  (temps managers + micro-inefficiences + rework, **à mesurer**)
- Coût non-service (pénalités + expedite + backlog) : **0,15–0,30 M€ / an**

**Total “sensible Praedixa” (ordre de grandeur)**

- Last-minute direct (HS + intérim + urgence) ≈ **0,86 M€ / an**
- - frictions + non-service ≈ **1,13 M€ à 1,36 M€ / an**

### 2.3 “Carte des décisions” (qui décide, quand, trop tard aujourd’hui)

**Hebdo (J+14)** — Responsable d’exploitation + planning + DAF (30–60 min/site ou régional)

- Ajuster enveloppe HS, quotas intérim, priorités clients, éventuellement sous-traitance.
- Aujourd’hui : décisions prises sans vision probabiliste des trous par créneau + sans coût complet.

**2×/semaine (J+7)** — planning + chefs d’équipe

- Commander intérim, rebasculer compétences, réaffecter polyvalents inter-zones.
- Aujourd’hui : commande trop tard → intérim “cher” / non dispo.

**Quotidien (J+3 et J+1)** — chefs d’équipe

- Arbitrer : HS de dernière minute, priorisation, cut-off.
- Aujourd’hui : mode pompier ; on subit le trou au lieu de le choisir.

### 2.4 Données disponibles typiques (exports, agrégés)

- **WFM / planning** : heures planifiées par site×date×shift×compétence (ou équivalent).
- **Réalisé** : heures pointées + HS (agrégées), absences (agrégées par motif si possible).
- **ERP/WMS/TMS** : volumes (commandes/lignes/colis), backlog, cut-off, retards.
- **Intérim** : heures consommées + coût + fournisseur + délai de commande.
- **Turnover** : entrées/sorties par semaine (agrégé site), ramp-up (si dispo en % prod).
- **Paramètres coûts** : coût horaire interne, majorations HS, coûts intérim, pénalités SLA, coût expedite (si suivi).

### 3.1 (1) Diagnostic + base de données canonique + raffinement continu

#### 3.1.A Objectif business (1 paragraphe) + KPI primaires

Construire une **source unique rejouable** “charge–capacité–coût” (agrégée), sans projet SI lourd, pour permettre (i) prévision fiable des ruptures, (ii) scénarios économiques auditables, (iii) preuve d’impact.
**KPI** : % créneaux couverts par la base (coverage), fraîcheur (D+0), taux d’erreurs data détectées, temps de correction mapping, rejouabilité 12–24 mois.

#### 3.1.B Inputs (données) : liste exhaustive “minimum viable” vs “premium”, granularité (site×créneau×compétence), fraîcheur, qualité attendue

**Grain canonique cible** : `site × date × shift × compétence` (compétence optionnelle en MVP)
**MVP (minimum viable)**

- Charge : volumes (ex. lignes, colis) par site×shift (D+0 à J+14) — **quotidien**
- Couverture planifiée : heures planifiées par site×shift (J+14) — **quotidien**
- Réalisé : heures réalisées, HS, absences totales (agrégé) par site×shift — **quotidien**
- Intérim : heures + coût total par site×jour (shift si dispo) — **hebdo**
- Calendrier : fériés/fermetures — **ponctuel**
- Coûts : coût horaire interne moyen, majoration HS, coût intérim moyen — **mensuel**
  **Premium**

- Absences : par grandes familles (maladie/AT/autre) — **quotidien**
- Turnover : entrées/sorties + “ramp-up” (prod relative) — **hebdo**

- Contraintes : caps HS, caps intérim, mobilité inter-sites, polyvalence (%), productivités par compétence — **mensuel**

### 3.2 (2) Prévision du risque de rupture de couverture (J+3/J+7/J+14)

#### 3.2.A Objectif business (1 paragraphe) + KPI primaires

Prédire, par site×créneau, le **risque** et la **sévérité** de sous-couverture à J+3/J+7/J+14 afin de déclencher des actions moins chères (intérim planifié, réallocation, ajustement service) et éviter le mode pompier.
**KPI** : recall sur ruptures “critiques”, précision des top alertes, calibration (fiabilité proba), faux positifs / semaine / site, gain € attribué par horizon.

#### 3.2.B Inputs (données) : liste exhaustive “minimum viable” vs “premium”, granularité (site×créneau×compétence), fraîcheur, qualité attendue

**MVP**

- Historique `charge`, `capacité planifiée`, `réalisé`, `absences totales`, `HS`, `intérim` (site×shift) sur 6–12 mois
- Calendrier (fériés), jour de semaine, saisonnalité simple
- Prévision charge à J+14 (même naïve : moyenne mobile / même jour-1 semaine)
  **Premium**
- Turnover hebdo + ramp-up (impact capacité effective)
- Indicateurs fournisseurs intérim (capacité/lead time)
- Signaux exogènes internes : promos, pics clients, cut-off exceptionnels, indisponibilités (matériel/quais)
  **Qualité**
- Timestamp fiable ; cohérence unités ; stable mapping site.

#### 3.2.C Pipeline & règles : étapes concrètes, contrôles qualité, versioning, traçabilité

1. Construction label : `gap_h = charge_h_est - capacite_dispo_h_est` (ou proxy via productivité)
2. Deux sorties :
   - `P(gap_h > 0)` (classification)
   - `E[gap_h | gap_h>0]` (sévérité)
3. Features interprétables :
   - charge récente, tendance, saisonnalité, backlog
   - capacité planifiée, absences attendues vs surprise
   - historique turnover, volatilité site, fiabilité intérim
4. Backtests glissants (rolling) par horizon
5. Calibration proba (isotonic/platt) + seuils d’alerte par criticité
6. Versioning modèle : dataset hash + code version + paramètres.

#### 3.2.E DoD (Definition of Done) : critères testables, backtests, calibration, taux d’alerte max, SLO

- Backtest 6 mois min :
  - Recall ruptures critiques (top X créneaux) **≥ 60%** en MVP, **≥ 70%** en premium (cible, à ajuster par bruit)
  - Calibration : écart proba prédite vs observée raisonnable (report Brier / reliability plot)
- Taux d’alertes : **≤ 8 alertes/semaine/site** (au-delà : seuils relevés + groupage)
- SLO calcul : refresh quotidien < 30 min après data ready ; disponibilité 99% ouvré.

#### 3.2.F Garde-fous & limites : quand ne pas recommander, comment éviter faux positifs, comment gérer données manquantes

- Si productivité (unités/heure) instable ou inconnue : on prédit d’abord **gap en heures** via proxies + bande d’incertitude.
- Si données absences trop bruitées : bascule vers modèle “volatilité site” + seuils plus conservateurs.
- Toujours afficher **incertitude** et **drivers**, pas un score opaque.
- Mode dégradé : si charge future non fiable → alerte uniquement sur capacité (absences/turnover) + mention “charge incertaine”.

#### 3.2.G Effort & automatisation : MVP (6–8 semaines) vs Premium (3–6 mois), tâches “agentisées” (automatisables), tâches humaines incompressibles

**MVP (6–8 semaines)**

- Automatisable : backtests rolling, calibration, génération rapport hebdo précision.
- Humain : définition “rupture critique”, validation seuils d’alerte, validation drivers.
  **Premium (3–6 mois)**
- Automatisable : tuning multi-sites, détection de régimes (pics), monitoring drift.
- Humain : revue mensuelle (Ops) des erreurs + ajustements process.

#### 3.2.H Valeur chiffrée (cas 10M€) : mécanisme (formule), ordre de grandeur, où se loge le gain (HS/intérim/organisation/service)

**Mécanisme** : plus tôt → options moins chères et plus disponibles.
**Formule gain (simplifiée)**
Gain ≈ Σ créneaux [ (coût_action_BAU − coût_action_anticipée) × %adoption ] − coût_erreurs
**Hypothèse** : 12 000 h HS/an et 18 000 h intérim/an, dont une partie “urgence” évitable.

- Si Praedixa convertit **10–20%** des heures “urgentes” en heures “planifiées” ou réallocation :
  - Gain HS : 0,299 M€ × (10–20%) = **30k–60k€/an**
  - Gain intérim+urgence : (0,540 M€ + 0,022 M€) × (8–15%) = **45k–85k€/an**
    **Où** : HS et intérim (coût direct) + service (moins de trous critiques).

### 3.3 (3) Moteur de scénarios économiques (exploitation data, pas du conseil)

#### 3.3.A Objectif business (1 paragraphe) + KPI primaires

Transformer un gap prédit en **options chiffrées comparables**, sous contraintes explicites, pour produire un arbitrage **coût vs service** utilisable par Ops/DAF, répétable et audit-ready.
**KPI** : % alertes avec ≥ 3 options valides, taux d’options “non dominées”, écart coût prévu vs réalisé, taux d’overrides (et raisons), délai moyen de décision.

#### 3.3.B Inputs (données) : liste exhaustive “minimum viable” vs “premium”, granularité (site×créneau×compétence), fraîcheur, qualité attendue

**MVP**

- Gap (heures) par site×shift×horizon
- Paramètres coût :
  - coût interne horaire (moyen)
  - majoration HS
  - coût intérim horaire
  - coût “dégradation service” (proxy : €/heure de backlog ou €/commande en retard)
- Contraintes simples : cap HS par site/jour ; dispo intérim (binaire)
  **Premium**
- Coûts par compétence / shift (nuit vs jour), minimum de commande intérim, pénalités SLA par client
- Courbes de productivité (ramp-up) + mobilité inter-sites + temps de trajet/coût
- Contraintes multi-périodes : dette fatigue, caps hebdo HS, règles internes (repos)
  **Qualité** : coûts validés DAF, datés, versionnés.

#### 3.3.C Pipeline & règles : étapes concrètes, contrôles qualité, versioning, traçabilité

1. Pour chaque alerte : générer espace d’actions (HS, intérim, réallocation, service)
2. Évaluer chaque option : **coût total attendu** + **service attendu** + **risque** (probabilité de non faisabilité)
3. Appliquer contraintes : éliminer options invalides
4. Construire frontière **non dominée** (Pareto coût/service)
5. Produire ranking mécanique (ex. coût minimal sous service ≥ seuil)
6. Log : paramètres utilisés + version coûts + contraintes + option retenue.

**Écran “Scénarios” pour un créneau**

- Gap (h) + incertitude
- Tableau options : coût€, service%, faisabilité%, hypothèses
- Frontière coût/service + recommandation (si autorisée)

## 4) MOTEUR ÉCONOMIQUE (détails concrets)

### 4.1 Liste des options (leviers)

1. **HS (heures supplémentaires)**
2. **Intérim** (planifié vs urgent si distinguable)
3. **Réallocation intra-site** (changement de zone/compétence)
4. **Réallocation inter-sites** (mobilité ponctuelle)
5. **Ajustement de service** (priorisation, décalage cut-off, backlog accepté)
6. **Sous-traitance / offre fournisseurs** (si applicable : co-packing, transport, overflow)

### 4.2 Paramètres, contraintes, formules (par option)

#### Option HS

- Paramètres coût : `c_int` (€/h), `maj_HS` (%)
- Contraintes : `cap_HS_shift` (h), règles repos, cap hebdo
- Formule :
  **Coût_HS = h_HS × c_int × (1 + maj_HS)**
  Service : couvre `min(h_HS, gap_h)`

#### Option Intérim

- Paramètres coût : `c_interim` (€/h), `premium_urgence` (%)
- Contraintes : lead time `LT`, capacité fournisseur `cap_interim`, minimum commande
- Formule :
  **Coût_int = h_int × c_interim × (1 + premium_urgence)**
  Faisabilité = 1 si `h_int ≤ cap_interim` et date ≥ aujourd’hui+LT sinon <1

#### Option Réallocation intra-site

- Paramètres : productivité relative `prod_ratio`, coût friction `c_friction` (€/h déplacée)
- Contraintes : polyvalence dispo, impact sur autre zone (ne pas créer un trou ailleurs)
- Formule (coût) :
  **Coût_realloc_intra = h_move × (c_friction)**
  (coût salarial déjà “dans le système”, on ne recompte pas ; on valorise la friction)
  Service : couvre `h_move × prod_ratio`

#### Option Réallocation inter-sites

- Paramètres : coût trajet/logistique `c_trajet` (€/pers/j), friction `c_friction_inter`
- Contraintes : distance, habilitations, disponibilité sur site source
- Formule :
  **Coût_realloc_inter = n_pers × c_trajet + h_move × c_friction_inter**
  Service : couvre `h_move × prod_ratio`

#### Option Ajustement service (backlog accepté)

- Paramètres : coût backlog `c_backlog` (€/h ou €/unité), pénalité SLA `c_SLA`
- Contraintes : backlog max, cut-off client, priorités A/B/C
- Formule :
  **Coût_service = h_non_couvert × c_backlog + pénalités_SLA(h_non_couvert)**
  Service : <100% (on accepte une dégradation contrôlée)

#### Option Sous-traitance / overflow

- Paramètres : coût unitaire `c_out` (€/h ou €/unité), minimum
- Contraintes : capacité sous-traitant, qualité, délais
- Formule : **Coût_out = q × c_out**

### 4.3 Sortie attendue : frontière coût/service + ranking non dominé

Pour chaque alerte : générer (coût€, service%) pour toutes options valides, retirer les options **dominées** :

- Une option A domine B si **coût_A ≤ coût_B** et **service_A ≥ service_B** (au moins une inégalité stricte).
  Puis produire :
- **Frontière Pareto** (ensemble non dominé)
- Ranking selon politique :
  - ex. “service ≥ 98% au coût minimal” (Ops)
  - ou “coût minimal sous contrainte SLA” (DAF)

### 4.4 Exemple complet (un créneau critique avec gap)

**Contexte (hypothèses)**

- Site : S_LYON, date 2026-02-10, shift PM
- Gap attendu : **38 h**
- Paramètres : `c_int=19,9 €/h`, `maj_HS=25%`, `c_interim=30 €/h`, `premium_urgence=0%` (car J+3), `cap_HS=30 h`, `cap_interim=50 h`, `c_backlog=60 €/h` (proxy coût retards + expedite)

**Options calculées**

1. **HS 30h + Intérim 8h** (respect cap HS)

- Coût = 30×19,9×1,25 + 8×30 = 746 + 240 = **986 €**
- Service = 100%

2. **Intérim 38h**

- Coût = 38×30 = **1 140 €**
- Service = 100% (cap OK)

3. **HS 30h + Service (backlog) 8h**

- Coût = 30×19,9×1,25 + 8×60 = 746 + 480 = **1 226 €**
- Service = (38-8)/38 = **79%** sur ce gap (dégradation contrôlée)

**Frontière non dominée (coût/service)**

- Option 1 : (986€, 100%) **non dominée** (meilleur coût à service max)
- Option 2 : (1 140€, 100%) **dominée** (plus cher, même service)
- Option 3 : (1 226€, 79%) **non dominée seulement si contrainte “pas d’intérim possible”** ; sinon dominée en pratique

**Recommandation mécanique (si politique = service max au coût min)** : **Option 1**.

## 6) COUVRIR (Capacity Hedging + exécution) — **la brique “killer”**

### 6.1 Objectif (1 paragraphe)

Transformer la prévision (risque de gap) en **capacité sécurisée** à coût maîtrisé : Praedixa construit dès **J+14/J+7** un **portefeuille de couverture** (réservations flexibles + leviers internes) puis **exerce / ajuste / annule** à **J+3/J+1** selon la matérialisation du risque.
Même si le **€/h d’intérim ne change pas** au dernier moment, la valeur vient de la réduction du **coût total attendu** : on augmente la probabilité d’être pourvu (fill rate) et on réduit les fallbacks coûteux (HS urgence / non-service / désorganisation).

### 6.2 Inputs (MVP vs Premium)

**MVP (suffisant pour POC)**

- Distribution de gap (ou à défaut : P(rupture) + gap P75/P90) par site×shift×date
- Catalogue “instruments” (par site) :
  - intérim : lead time (jours), cap (h/j), conditions d’ajustement (annulation jusqu’à T-24/T-48), coût €/h
  - HS : cap HS/shift, majoration, règles simples
  - service : coût proxy backlog / pénalité (même grossier), backlog max

- Indicateurs fournisseurs (même approximatifs) : fill rate historique, no-show (si dispo)

**Premium**

- Instruments multi-fournisseurs, SLA, caps par compétence/shift, minimum de commande
- Réallocation inter-sites (mobilité, frictions, temps trajet)
- Modèle d’incertitude complet (scénarios de gap) + optimisation risque (CVaR / worst-case)
- Connecteurs d’exécution (ticketing / demande intérim / validation Ops)

### 6.3 Output (ce que l’utilisateur voit)

Pour chaque créneau critique :

- **Plan de couverture recommandé** (portefeuille) : ex. “Réserver 20h intérim (annulable T-48) + pool HS 10h + accepter 5h backlog”
- **Coût fixe** (prime/effort de réservation si applicable) + **coût variable** (si exercé)
- **Service garanti** (cible) + **risque résiduel** (probabilité de rupture après hedge)
- **Règles d’exécution** : quand on exerce, quand on annule, et sous quelles contraintes (audit-ready)

### 6.4 Mécanisme “A + B” (preuve DAF)

**A.** Le coût attendu dépend surtout de la probabilité d’être pourvu, pas du tarif :
[
\mathbb{E}[coût]=P(\text{pourvu})\cdot C*{\text{intérim}}+(1-P(\text{pourvu}))\cdot C*{\text{fallback}}
]
avec (C\_{\text{fallback}}) = HS urgence + désorganisation + non-service.
**B.** Réserver tôt une capacité flexible augmente (P(\text{pourvu})) et réduit le recours au fallback : on remplace l’urgence par une flexibilité maîtrisée, donc le **coût total attendu baisse** même si le €/h est identique.

### 6.5 KPI (très vendables)

- **Hedge Coverage Ratio** : % des créneaux critiques couverts par une réserve flexible
- **Emergency Avoidance** : heures “urgentes” évitées (HS/intérim) vs BAU
- **Option Efficiency** : (coût urgence évité) / (coût de couverture)
- **Exercise Accuracy** : % de réserves exercées “à bon escient”
- **Service-at-Risk** : probabilité de passer sous SLA après hedge

### 6.6 Pourquoi les agences d’intérim acceptent de se brancher

- Plus de missions pourvues (visibilité J+7/J+14) → CA ↑
- Coût de sourcing ↓ (moins de pompier) → marge ↑
- Stabilité (règles d’ajustement/annulation) → moins d’aléas
- KPI & SLA (fill rate, no-show, lead time) → amélioration continue + avantage concurrentiel
- Potentiel de volume récurrent (fournisseur “préféré”)

### 6.7 DoD (Definition of Done) — MVP POC

- Plan de couverture généré pour 100% des alertes critiques (au moins 2 instruments)
- Exécution/annulation tracée dans le decision log
- Proof pack : comparaison BAU vs hedge (urgence évitée, service stabilisé, efficacité de couverture)
- Cap sur-alerte maintenu (ex. ≤ 8 alertes/semaine/site) via groupage + seuils

## 5) PROUVE DE VALEUR & ATTRIBUTION (0% / 100% / réel)

### 5.3 Protocole d’attribution pragmatique

**Approche principale : holdout sites (recommandée ici)**

**Fallback : alternance temporelle**

- Tous sites : 1 semaine ON / 1 semaine OFF (ou 2 jours ON / 2 jours OFF) sur créneaux comparables.

8. Plan d’action : ajustements seuils/coûts/contraintes + responsables + date

3. Proposition pilote “service cheval de Troie” (forfait forecasting staff) + plan 8 semaines
4. Pilote cheval de Troie → proof pack → conversion abonnement full package
   **Cycle** : 6–10 semaines jusqu’au pilote signé ; 10–14 semaines jusqu’à conversion (hypothèse).
   **Qualification (go/no-go)** :

- Exports charge + couverture + réalisé (même incomplets) disponibles
- Intérim et/ou HS significatifs
- Sponsor Ops + sponsor DAF identifiés
- Acceptation du decision log

### 7.3 Plan livraison: de 0 à 3 pilotes, puis rollout; ressources nécessaires

**Ressources (12 mois) – scénario base**

- 1 CEO/PM (produit+vente)
- 1 data/ML ops (prévision + MCO)
- 1 data engineer (pipelines + qualité)
- 0,5 CSM/ops (à partir de 3–4 clients payants) (hypothèse)

**Plan**

- M1–M2 : 1 pilote
- M3–M5 : 2e pilote + conversion #1
- M6–M8 : 3e pilote + conversions
- M9–M12 : rollout multi-sites + industrialisation connecteurs légers

### 7.4 Unit economics (hypothèses): CAC, ACV, marge brute, infra, support/implémentation, LTV, payback

**Hypothèses (à valider)**

- ACV Pro : 90 k€/an
- Marge brute SaaS (hors implémentation) : 80–90% (infra modérée)
- Coût implémentation/pilote : 12–20 k€ (temps équipe) ; diminuant avec templates
- CAC (fully loaded, 12 mois) : 35–70 k€ (selon volume outbound + cycles) (hyp.)
- Churn annuel logo : 8–15% (hyp.)
  **LTV (simplifiée)** : LTV ≈ ACV × marge brute / churn
- Exemple : 90k × 85% / 12% ≈ **638 k€**
  **Payback CAC** : CAC / (ACV×marge brute)
- 50k / (90k×0,85) ≈ **0,65 an** (≈ 8 mois)

### 7.5 Forecast: 3 scénarios (conservateur / base / agressif) sur 12 mois + cap vers 36 mois

**Hypothèse prix moyen** : 90 k€ ACV ; pilote 35 k€ (partiellement crédité)

**12 mois**

- Conservateur : 2 pilotes, 1 conversion → ARR fin année ~90 k€
- Base : 3 pilotes, 2 conversions → ARR fin année ~180 k€
- Agressif : 5 pilotes, 4 conversions → ARR fin année ~360 k€

**Cap 36 mois (ordre de grandeur)**

- Base : 25–35 clients Pro/Ent, ARR ~2,5–4,0 M€ (si industrialisation + faible churn)

### 7.6 Principaux risques (tech/data/vente) + mitigations concrètes

- Data hétérogène → templates d’import + DQ gates + mode dégradé
- Adoption faible → alerting capé + rituels + decision log obligatoire
- Scénarios “contestés” → coûts versionnés DAF + transparence hypothèses
- Prouver attribution → holdout by design + proof pack mensuel
- Cycle de vente long → pilote forfait packagé + crédit partiel

## 8) FEASIBILITY “REALITY CHECK”

### 8.1 Ce qui est faisable vite (MVP 6–8 semaines) vs ce qui est dangereux (et pourquoi)

**Faisable vite (MVP)**

- Base canonique + rejouabilité 12 mois (exports agrégés)
- Prévision risque simple & utile (modèle interprétable + seuils) à J+3/J+7/J+14
- Scénarios v1 (HS/intérim/service) avec coûts moyens + contraintes simples
- Decision log v1 + proof pack fin de pilote

**Dangereux (à éviter en MVP)**

- Optimisation multi-sites “globale” type solveur complet (trop de contraintes implicites)
- Promesse de précision fine sans historique propre (risque de faux positifs)
- Calcul coût “par client” sans pénalités SLA fiables (risque de contestation DAF)
- Connecteurs SI profonds multi-outils (dérive projet SI)

### 8.2 10 pièges classiques + garde-fous

1. Trop d’alertes → **cap alertes/site/semaine**, groupage
2. Données instables → **DQ gates** + mode dégradé
3. Coûts non alignés DAF → **atelier coûts + versioning**
4. Réallocation crée un trou ailleurs → contrainte “no negative spillover”
5. Intérim supposé disponible → intégrer **capacité/lead time**
6. Service non mesuré → définir **proxy** (backlog, cut-off)
7. Decision log ignoré → champ obligatoire minimal + relances
8. “Score opaque” → drivers + incertitude affichés
9. Pas de contrefactuel → holdout dès le départ
10. Mélanger planning individuel → rester **agrégé** (contrainte produit)

### 8.3 Liste des “données minimum” qui, si absentes, rendent le POC non concluant (avec contournements)

**Bloquantes (sans contournement crédible)**

- Couverture planifiée (heures) par site×shift (au moins sur 8–12 semaines)
- Réalisé (heures + HS) par site×shift (ou par jour si shift impossible)
- Charge (volumes) par site×shift (ou proxy : expéditions/jour + cut-off)

**Fortement souhaitables (contournables avec mode dégradé)**

- Intérim heures+coûts → contournement : coût moyen + extraction factures mensuelles
- Absences agrégées → contournement : volatilité site + résidus sur réalisé vs plan
- Backlog/service → contournement : proxy via retards cut-off / expéditions J+1

**Règle Go/No-Go POC (simple)**

- Si ≥ 3 semaines sur 8 ont >10% de créneaux manquants (charge ou couverture) sur sites pilotes → POC “preuve” non concluant, on fait d’abord “data enablement”.
