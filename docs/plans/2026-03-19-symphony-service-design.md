# Symphony Service Design For Praedixa

## Intent

Praedixa needs a first-class service that can execute repository work through Codex with the same discipline already applied to the product and data planes: typed configuration, strict boundaries, explicit observability, and reproducible runtime behavior. The implementation should stay outside `app-api-ts` and `app-api` because it is neither product HTTP traffic nor Data/ML orchestration. The recommended placement is a dedicated Node/TypeScript runtime named `app-symphony`. This keeps the service close to the existing monorepo tooling, allows direct reuse of `@praedixa/telemetry`, and avoids mixing agent-automation concerns into the production customer APIs. The design follows the `docs/specs/SPEC.md` contract while adding a Praedixa-specific harness profile that emphasizes git worktrees, per-issue runtime metadata, and fail-closed execution around secrets, paths, and approvals.

The service is configured by `WORKFLOW.md` in the repository root by default, with a CLI override for alternate workflow files. The front matter remains the source of truth for tracker selection, polling, workspace behavior, hooks, agent concurrency, and Codex runtime policy. Praedixa adds a `harness` extension block to support reproducible worktree creation and per-issue metadata without polluting the core spec. This extension is intentionally small: worktree strategy, base ref, branch prefix, optional copied env files, number of reserved ports, and log directory policy. Everything else stays in shell hooks so the repository can evolve the environment without modifying the orchestrator itself.

## Architecture

`app-symphony` is split into small modules. `workflow.ts` loads and validates `WORKFLOW.md`, parses YAML front matter, and renders strict Liquid templates. `config.ts` exposes the typed runtime view and handles env indirection, defaults, and path normalization. `linear-client.ts` owns GraphQL pagination and normalization into a stable issue model. `workspace-manager.ts` enforces sanitization, root containment, hook execution, and harness metadata creation. `harness.ts` is responsible for git worktrees, reserved ports, env copying, and `.symphony/` manifests inside each issue workspace.

`codex-app-server.ts` wraps the JSON-RPC protocol over stdio. It performs `initialize`, `initialized`, `thread/start`, `turn/start`, optional `configRequirements/read`, and streaming event handling. It auto-resolves command and file approvals according to the configured policy, hard-fails `tool/requestUserInput`, and exposes an optional dynamic tool named `linear_graphql` backed by the active Linear credentials. `orchestrator.ts` owns claims, retries, runtime metrics, stall detection, reconciliation, and dispatch ordering. `status-server.ts` exposes `/`, `/api/v1/state`, `/api/v1/:issueIdentifier`, and `/api/v1/refresh`. `index.ts` wires CLI startup, workflow watch/reload, startup validation, terminal workspace cleanup, and graceful shutdown.

This structure mirrors the repository’s existing style: strong separation of concerns, explicit boundaries, and testable modules. It also keeps every source file comfortably inside the repo guardrails on file and function size.

## Harness And Safety Model

Praedixa’s harness should be strong by default because this service will run long-lived coding agents against a privileged monorepo. Each issue receives a deterministic workspace rooted under `workspace.root`, plus a `.symphony/` directory containing `workspace.json`, reserved ports, copied env file inventory, session log paths, and recent run metadata. The preferred strategy is `git worktree add`, not ad hoc directory copies. That gives the agent a real repository with correct history while keeping issue execution isolated. Worktree branch names use a sanitized `branch_prefix + issue identifier`, and cleanup uses `git worktree remove` only for terminal issues or explicit operator action.

Safety remains fail-closed. The service validates that every workspace path is inside the configured root, every hook executes from that workspace, and every Codex thread/turn is launched with that workspace as `cwd`. Secrets are never injected via CLI args. If env files are copied into a worktree, the copy list must be explicit in `WORKFLOW.md`; there is no implicit broad secret sync. `linear_graphql` reuses the configured Linear API key from the orchestrator process and never asks the agent to discover raw tokens on disk. `tool/requestUserInput` is treated as a hard failure to avoid stuck runs. Approval and sandbox posture are documented at startup and surfaced in logs and HTTP state so operators can confirm the service is running under the intended trust model.

## Verification Plan

The implementation should ship with focused unit coverage around the fragile boundaries: workflow parsing and reload, Liquid strict-render failures, Linear pagination and normalization, workspace root containment, hook timeout handling, retry scheduling, stall detection, Codex event parsing, approval handling, dynamic tool calls, and HTTP state rendering. We do not need live Linear credentials or a live Codex server for core conformance tests; those are isolated behind fakes. A smaller integration profile can optionally exercise a real `codex app-server` process and real Linear GraphQL when secrets are available.

Repository integration also needs to be explicit. Root scripts should expose `pnpm dev:symphony`, `pnpm build:symphony`, and `pnpm test:symphony`. The top-level TypeScript project references must include the new app. `scripts/README.md`, `docs/ARCHITECTURE.md`, and `infra/README.md` should document the service as an internal automation plane, distinct from the product APIs and from the Python medallion orchestrator. That gives Praedixa a versioned, inspectable automation service instead of a pile of ad hoc agent runner scripts.
