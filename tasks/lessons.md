# Lessons

- 2026-03-27: Quand un hero video doit etre encore simplifie, commencer par retirer les companion pills secondaires (`QSR OPS`, `30j`, badges stats) avant de toucher a la typo ou aux CTA; ce sont souvent les premiers parasites visuels.
- 2026-03-27: Quand le contraste du header pose probleme au-dessus d'un hero `hero-pulsor`, corriger d'abord les regles globales `body:has(.hero-pulsor) header[data-scroll-surface="top"] ...` avant de toucher aux composants de nav; c'est la vraie source de theme pour l'etat top.
- 2026-03-27: Quand l'utilisateur veut remettre de la video dans le hero tout en le rendant plus "degage", ne pas poser la video sous la meme structure qu'avant; repartir d'une composition minimale et retirer les panneaux, rails et preuves redondantes au-dessus de la ligne de flottaison.
- 2026-03-27: Quand l'utilisateur demande de retirer le bloc sous le hero, ne pas conserver par reflexe une section "answer-first" post-hero; si ce resume reste utile, il doit etre absorbe par le hero lui-meme et non impose comme un bloc supplementaire.
- 2026-03-27: Quand un utilisateur recadre un hero comme "trop de gros blocs rectangulaires", retirer d'abord les conteneurs lourds et reconcentrer la preuve en lignes, pills ou elements flottants; la sensation premium depend d'abord de la silhouette generale, pas du detail interne des cartes.
- 2026-03-26: Quand un setup MCP Figma doit permettre l'ecriture dans Figma, ne pas s'arreter a la lecture du fichier ou au statut `enabled`; verifier explicitement que les outils de write (`use_figma`, `create_new_file`, `generate_figma_design`) sont bien charges dans la session active.
- 2026-03-26: Quand on ajoute ou authentifie de nouveaux outils MCP/skills pendant une conversation Codex, ne pas supposer que le thread courant recharge son schema d'outils; verifier sur un NOUVEAU thread avant de conclure que l'integration ne marche pas.
- 2026-03-23: Quand un app Next.js de dev doit etre ouverte via l'IP LAN du poste, ajouter aussi ces IPv4 locales a `allowedDevOrigins`; sinon Next bloque `/_next/*` et le symptome peut ressembler a un bug auth.
- 2026-03-23: Quand une app Next OIDC tourne en local sur une IP privee ou un alias loopback, garder le meme host pour les cookies de login, le `redirect_uri`, le callback et les controles same-origin; melanger IP reseau et `localhost` casse le round-trip auth.
- 2026-03-23: Quand l'utilisateur demande explicitement de couper GitHub Actions, desactiver les definitions de workflow elles-memes et arreter de chasser les rouges un par un.
- 2026-03-23: Quand on corrige une GitHub Action ou un workflow, verifier toutes les refs restantes dans le repo et confirmer le SHA de la PR distante avant d'annoncer le fix; un vieux run rouge peut masquer les vrais bloqueurs encore actifs.
- 2026-03-22: Quand un repo versionne `@playwright/test`, lancer toujours les scripts et gates E2E via `pnpm exec playwright`; un binaire global plus vieux peut casser toutes les specs avec un faux symptome de config.
- 2026-03-22: Quand un pack Playwright large devient heterogene ou en drift, ne pas le laisser bloquer tous les commits; faire du lot bloquant une suite critique reduite et production-like (`next start` apres build), puis garder le sweep complet en manuel jusqu'a realignement.
- 2026-03-22: Quand une surface webapp est reouverte par un drapeau runtime (`messagingWorkspace`, `userPreferencesPersistence`), rerun tout de suite les E2E de la page concernee; les unit tests mockes ne suffisent pas a prouver que le runtime reel est encore ouvert.
- 2026-03-22: Quand un monorepo TypeScript a plusieurs apps et packages, ne pas se contenter d'un unique `tsc --build` cache dans un gate profond; ajouter un agregateur de typecheck par projet au `pre-push` pour remonter toutes les erreurs de type d'un coup.
- 2026-03-22: Quand on versionne un contrat runtime genere, ne pas le limiter aux secrets et a l'origin frontend; inclure aussi les variables runtime non secretes obligatoires (`NEXT_PUBLIC_API_URL`, OIDC issuer/client/scope, reglages auth front), sinon un drift de deploiement reste invisible.
- 2026-03-22: Quand l'utilisateur demande explicitement une grosse passe `clean-code` avec plusieurs agents en parallele, lancer tout de suite des sous-taches focalisees par zones de responsabilite au lieu de garder toute la passe sur un seul thread.
- 2026-03-22: Quand un workflow d'onboarding expose plusieurs familles d'actions metier, ne pas forcer un format cockpit unique; si l'operateur doit avancer pas a pas, le bon paradigme est un assistant par etapes et non une grande colonne de panneaux.
- 2026-03-22: Quand un ecran operateur melange lancement, selection et execution dans la meme densite visuelle, le probleme n'est pas du polish mais de l'architecture d'information; il faut separer explicitement ces zones avant d'ajouter du style.
- 2026-03-22: Quand une surface admin reconsomme le runtime integrations local, rebrancher explicitement le meme gate local et le meme bootstrap `app-connectors` que `/config`; sinon l'onboarding se remet a spammer des `500` garantis alors que le backend metier lui-meme est sain.
- 2026-03-22: Quand une feature backend depend d'un nouveau secret runtime, le brancher dans le code ne suffit pas; il faut aligner dans le meme diff les loaders `.env.local`, l'inventaire des secrets, les scripts de config/deploy et le preflight cloud.
- 2026-03-22: Toute nouvelle route API publique ou webhook doit recevoir sa policy d'exposition (`exposure-policy.ts`) dans le meme diff, sinon le runtime renvoie `EXPOSURE_POLICY_MISSING` meme si le handler et les tests de route existent.

- Quand un garde-fou anti-boucle auth doit survivre a un rebond immediat, ne pas le stocker en `sessionStorage` de facon collante; preferer un marqueur URL ou temporel qui expire vite pour ne pas casser les refreshs et redemarrages suivants.
- Quand une page de login affiche un code d'erreur derive de l'URL, verifier si cet etat peut etre stale apres retour du provider et auto-reconcilier une fois avant de laisser croire que la panne est toujours active.
- Quand une policy runtime OIDC est consommee par `app-admin` et `app-webapp`, la garder dans un helper partage unique plutot que dans deux copies locales; sinon les regles de confiance et de revocation divergent silencieusement.
- Quand un login admin local reussit mais finit en `reauth=1`, verifier tout de suite l'issuer/JWKS reels de `app-api-ts`; un vieux `AUTH_ISSUER_URL` live dans le shell ou `app-api/.env` peut faire rejeter les JWT Keycloak locaux apres le callback.
- Quand l'utilisateur recadre explicitement sur le dev local, verifier tout de suite les `.env.local` et l'issuer/runtime reels avant de poursuivre sur une logique purement prod.
- Quand un flux OIDC local utilise un IdP sur loopback, verifier si le garde-fou de confiance refuse l'HTTP localhost lui-meme avant de diagnostiquer TLS/certificats/endpoints.
- Quand un script shell sous `scripts/keycloak` vise un contrat repo-owned sous `infra/auth`, resoudre le chemin depuis `REPO_ROOT` plutot que depuis `SCRIPT_DIR/../infra`, sinon le helper pointe sur `scripts/infra/...` et casse au runtime.
- Quand le runtime porte deja la compatibilite legacy, garder les migrations de contrat en DDL plat et supprimer les blocs de reconciliation du fichier de migration lui-meme; sinon le parseur statique et le contrat cible divergent inutilement.
- Quand un parseur SQL ne comprend pas les blocs proceduraux d'une migration de base, revenir a du DDL plat et a des `CAST(...)` standards plutot que d'introduire `DO $$` ou `IF NOT EXISTS` qui cassent le parseur.
- Quand un helper de seed exige un tenant concret, ne pas conserver un `UUID | None` fictif juste pour l'ergonomie; rendre l'UUID explicite dans le CLI et dans la signature pour que Pylance et le runtime racontent la meme histoire.
- Quand une signature de seed/provision perd un keyword, retirer le parametre stale de tous les call sites dans la meme passe et revalider le fichier, sinon Pylance continue de signaler une API fantome.
- Quand un helper Sonar ne sert plus qu'a porter un pattern bruité, verifier d'abord s'il est encore appele; si non, supprimer le helper plutot que de le cosmetiser.
- Quand Sonar demande l'operateur oppose pour une comparaison de bornes, ecrire directement la forme positive correspondante au lieu de laisser une negation implicite.
- Quand une fonction d'enrichissement ajoute a la fois du cache, des requetes externes et des calculs par ligne, extraire d'abord les helpers "cache", "fetch" et "row" pour garder l'orchestration lisible.
- Quand Pylance ne comprend pas les exports NumPy directs, centraliser les appels dans un petit module adapter local plutot que de multiplier les imports fragiles partout dans les scripts.
- Quand un predicteur numerique est represente comme `float | None`, filtrer le `None` avant de construire un tableau NumPy plutot que de compter sur un `any(...)` pour la narrowing du type.
- Quand un helper de parsing est partage entre plusieurs structs proches, typer la signature sur le contrat reel commun au lieu de le refermer sur la premiere source appelee.
- Quand un clamp ou une aggregation a plusieurs niveaux devient trop complexe, extraire d'abord la version "un groupe" ou "une ligne" puis garder le wrapper orchestral tres mince.
- Quand on extrait un helper depuis une fonction a fenetre glissante, propager explicitement le parametre de fenetre du call site au helper au lieu de le figer par inadvertance.
- Quand Pylance signale un symbole NumPy fragile dans un script, preferer un chemin plus direct et typable (ou un fallback natif comme `abs` / `ValueError`) plutot que d'accumuler des aliases imports qui restent suspects.
- Quand un script unique devient trop gros pour etre relu ou typed proprement, le meilleur refactor est de le casser en modules par responsabilite et de garder un facade publique stable pour les tests existants.
- Quand l'utilisateur rappelle que le repo n'est pas en production et veut supprimer le legacy, ne pas sur-preserver les doublons, wrappers test-only ou docs faibles "au cas ou": privilegier le nettoyage agressif tant que les references repo et les verifications restent vertes.
- Quand l'utilisateur repointe explicitement un skill ou son contenu, verifier les chemins et artefacts exacts qu'il fournit avant de conclure qu'un outil annexe est absent ou indisponible.
- Quand l'utilisateur veut que les modeles s'entrainent sur le site, ne pas s'arreter aux bots de search: autoriser explicitement les tokens de training documentes (`GPTBot`, `ClaudeBot`, `Google-Extended`) sur le corpus public voulu et garder le blocage uniquement sur le technique/prive.
- Quand l'utilisateur veut un anti-scraping fort mais exige aussi un GEO/LLM crawl optimise, ne pas bloquer par reflexe les crawlers LLM sur les surfaces P2/P3; distinguer immediatement les pages publiques sacrifiables des APIs, assets signes et routes techniques a fermer.
- Quand l'utilisateur demande de "ne pas s'arreter" ou de faire le travail "a 100%", prolonger tout de suite l'execution sur le prochain chaînon productible au lieu de cloturer sur un verdict ou un demi-fix.
- Quand l'utilisateur demande de "continuer le PRD" ou de "terminer le TODO", produire directement les artefacts de fermeture manquants et les brancher au `TODO`, au lieu de s'arreter a des docs de cadrage meta ou a une simple recommandation.
- Sur un chantier monorepo long avec plusieurs sections ouvertes du `TODO`, lancer tout de suite plusieurs agents en parallele pour cartographier les options de fermeture, au lieu de faire une exploration sequentielle trop lente.
- Quand un gate local long reste bloque sur une seule suite, remonter vite au user le blocage exact, l'etape suivante et l'horizon de cloture, au lieu d'attendre la fin complete du pipeline.
- Quand un script ops local a besoin d'un secret deja present dans les `.env.local` standard du repo, verifier et autocharger ces fichiers avant de demander a l'utilisateur de reexporter la variable a la main.
- Quand un utilisateur demande de supprimer des comptes fake/demo, ne pas s'arreter au fait de fermer le chemin de creation: inventorier et purger aussi les identites deja presentes en live, ou expliciter tout de suite le blocage restant.
- Quand l'utilisateur demande si "ca fonctionne", repondre d'abord depuis la realite de production; un flux seulement valide en local ou dans le worktree n'est pas une reponse suffisante.
- Quand un hook pre-push remonte une CVE patchable sur `next`, traiter le bump de patch dans toutes les apps Next du monorepo avant de pousser, au lieu d'essayer de contourner le gate.
- Quand un nouveau symbole est exporte depuis `@praedixa/shared-types`, rebuilder tout de suite le package avant de relancer les apps/tests consommatrices, sinon elles restent branchees sur un `dist/` stale et on debugge un faux symptome.
- Quand l'utilisateur dit qu'un fix admin "ne fonctionne toujours pas", reproduire immediatement le flux complet en navigateur avec la session/runtime reels et remonter l'erreur reseau exacte avant de conclure que le probleme est regle.
- Quand l'utilisateur dit explicitement de ne pas m'arreter tant qu'il reste du travail utile, ne pas envoyer un simple close-out de wave: enchaîner directement la wave suivante jusqu'a un vrai blocage ou un repo sensiblement vide de gains clairs.
- Quand l'utilisateur parle de "mot de passe envoye" sur un flux d'invitation, verifier d'abord si le produit cree reellement un compte/invite ou seulement l'organisation; ne pas confondre creation de tenant et provisionnement d'acces.
- Quand un flux admin "supprime un client test", verifier qu'il purge aussi les identites Keycloak provisionnees pour ce tenant; sinon la suppression reste partielle et la recreation peut tomber en conflit.
- Quand un client test est recréable par email, ne pas se limiter aux `users.auth_user_id`: purger aussi les comptes Keycloak orphelins retrouvables par email et attribut `organization_id`, sinon un ancien tenant supprime peut encore bloquer la recreation.
- Quand un `POST /users` Keycloak repond `409`, ne pas l'interpreter automatiquement comme un conflit d'email: verifier aussi les collisions de `username` et les users legacy sans attributs canoniques, sinon le rattrapage "orphelin par email" peut sembler corriger le bug tout en laissant le cas reel intact.
- Quand `pnpm dev:api` doit supporter des mutations admin qui provisionnent dans Keycloak, autocharger aussi `KEYCLOAK_ADMIN_USERNAME` et `KEYCLOAK_ADMIN_PASSWORD` au meme niveau que `DATABASE_URL`; sinon l'API demarre mais le provisioning tombe en `503` au premier usage.
- Quand l'utilisateur dit qu'il n'a toujours pas les logs locaux, verifier si les wrappers `dev:*` tournent en background avec redirection silencieuse vers un fichier, et remettre un mode terminal-attache par defaut pour le debug.
- Quand un proxy local remonte `proxy.upstream_failed` vers `localhost`, verifier d'abord que le backend ecoute vraiment sur le port attendu; un PID `pnpm/tsx` encore vivant ne prouve pas que le serveur HTTP est monte.
- Quand Safari ou Playwright remonte une violation CSP inline sur une page Next, verifier tout de suite les providers comme `next-themes`: leur bootstrap inline a besoin du nonce propage depuis le layout serveur.
- Quand une mutation admin persiste un audit ou un historique avec l'acteur courant, verifier sur la vraie base locale que le `sub` OIDC se mappe bien a `users.id`; sinon persister un fallback `auth_user_id` separe au lieu de laisser une FK casser toute la transaction.
- Quand un endpoint admin compose plusieurs lectures SQL pour un dashboard, verifier les types et les noms de tables contre le schema reel local: un join `uuid = varchar` ou une table legacy fantome comme `employees` peut faire tomber tout le payload `overview`.
- Quand l'utilisateur remet en cause la "completude" d'un workflow admin, comparer explicitement le blueprint cible aux commandes et pages vraiment exposees; une page BPM avec des taches completable ne suffit pas a conclure que tout le process metier est couvert.
- Quand une premiere migration est faite sur une app mais que l'utilisateur elargit ensuite explicitement le perimetre a "tout" le monorepo, repartir tout de suite sur une passe globale et documenter ce nouveau scope au lieu de rester implicite sur le sous-perimetre precedent.
- Quand un cluster de fichiers front remonte des findings Sonar JSX bruités, privilegier des JSX plus explicites (pas de fragment court inutile, pas de texte inline ambigu avant un tag) avant de conclure a un faux positif outil.
- Quand l'utilisateur demande qu'une implementation colle exactement a un spec repo-owned, faire d'emblee un sweep de conformite exhaustif spec-vers-code et corriger les ecarts avant de revenir, au lieu d'avancer par petites iterations reactives.
- Quand un lot `components/` est explicitement decoupe par exclusions, ne pas retoucher les fichiers reserves a d'autres lots; verifier le perimetre avant chaque edit pour eviter les collisions.
- Quand une page admin locale pointe encore vers des routes stub ou vers un runtime optionnel non configure, couper le fetch avant le mount et rendre un message fail-close explicite au lieu de remplir la console de `503/500` attendus.
- Quand une seule surface d'un workspace admin devient enfin persistante, scinder les feature gates par endpoint pour rouvrir uniquement cette surface sans rebrancher ses voisins encore en stub.
- Quand une operation destructive admin ne doit viser que des tenants de test, persister un flag explicite `isTest` a la creation et le faire verifier cote serveur; ne jamais deduire ce statut d'un slug, d'un email ou d'une convention implicite.
- Quand j'ajoute une migration Alembic nommee, garder un `revision` court compatible avec la taille de `alembic_version.version_num`; un identifiant trop long peut faire echouer l'upgrade apres le DDL.
- Quand j'ajoute un nouveau runtime interne qui depend de secrets locaux, autocharger explicitement son `.env.local` et son `.env` versionnes hors git au lieu de supposer que l'utilisateur a exporte les variables dans son shell.
- Quand un contrat repo-owned comme `WORKFLOW.md` est consomme depuis un sous-package lance via `pnpm --dir`, ne jamais resoudre le chemin par rapport au `cwd` du package seul; remonter jusqu'a la racine qui porte vraiment le contrat.
- Quand j'autochage un `.env` pour un sous-package lance via `pnpm --dir`, ne pas construire les chemins a partir de `process.cwd()` seul; utiliser aussi le vrai dossier du package, sinon `app-symphony/.env` devient `app-symphony/app-symphony/.env` et les secrets restent invisibles.
- Quand l'utilisateur recentre explicitement la demande sur un artefact de contenu precis, ne pas repartir sur le debug du runtime ou sur la derniere erreur en cours; livrer d'abord l'artefact demande.
- Quand je poursuis une passe d'audit ou de hardening sur un chantier long, rappeler explicitement dans mes updates le perimetre exact en cours pour eviter que l'utilisateur doive demander sur quoi je travaille.
- Quand l'utilisateur corrige le modele commercial d'une offre, repercuter immediatement cette verite dans tout le funnel (hero, parcours, CTA, docs, tests) au lieu de ne corriger qu'un seul bloc visible.
- Quand l'utilisateur recadre le messaging de la landing, propager ce framing dans tous les blocs homepage visibles (hero, comparatif, preuve, FAQ, CTA) au lieu de laisser survivre l'ancien vocabulaire dans les sections voisines.
- Quand un audit append-only reference une entite supprimable, ne pas garder une FK vivante `ON DELETE SET NULL` vers cette entite; stocker une reference historique simple ou la suppression restera bloquee en base.
- Quand un utilisateur signale que les tests/gates locaux saturent sa machine, repasser immediatement les commandes repo-wide de couverture et E2E en mono-worker par defaut, puis ne laisser la parallelisation qu'en opt-in explicite.
- Quand le repo versionne deja un runner de deploiement cible, verifier d'abord cette cible reelle avant d'inferer une autre plateforme a partir d'un script applicatif residuel.
- Quand j'injecte un secret dans une session TTY pour une action ops, couper l'echo terminal (`stty -echo`) ou preferer une pipe non interactive afin que le secret ne ressorte jamais dans la sortie de session.
- Quand un script shell genere un JSON via un sous-process Python/JQ, exporter explicitement les variables attendues avant l'appel; des variables shell non exportees donnent un faux "succès" avec une config partielle.
- Quand un provider deploiement transporte une commande container comme liste d'args, ne pas supposer qu'un script multi-ligne restera un seul argument; compacter ou versionner le bootstrap autrement avant de toucher le runtime live.
- Quand l'utilisateur veut que j'aille jusqu'au bout d'un chantier evident, ne pas redemander a chaque chaînon si je dois poursuivre; executer directement toute la passe utile puis ne remonter qu'un vrai blocage, un risque cache ou un resultat final.
- Quand je lance plusieurs actions `kcadm` en parallele, ne jamais partager le `~/.keycloak/kcadm.config` par defaut; utiliser un `--config` isole par workflow sinon les locks et sessions se polluent.
- Quand je compte sur `kc.sh start --import-realm`, copier l'export sous un nom `*-realm.json` dans `/opt/keycloak/data/import/`; un fichier mal nomme peut etre ignore au boot et faire disparaitre le realm apres redeploy.
- Quand un patch multi-fichiers commence a diverger du contexte attendu, relire l'en-tete exact du fichier avant de reposer le diff, et pour les lints sur une liste de fichiers, preferer `xargs` ou une expansion verifiee plutot qu'une variable multiline brute.
- Quand un rendu React derive du contenu a partir d'un callback local, declarer le callback avant les constantes JSX derivees pour eviter les erreurs de portee / TDZ dans le meme fichier.
- Quand j'execute `pnpm --filter @praedixa/admin exec eslint`, utiliser les chemins relatifs au package `app-admin` (par exemple `components/...`) et non les chemins racine du repo, sinon je valide un faux echec de chemin plutot que le code.
- Quand l'utilisateur pointe un warning statique precis (Sonar, lint, typecheck), traiter ce signal comme un vrai lot de correction source et filtrer tout de suite le bruit `.next` / code genere avant d'estimer le scope.
- Quand un service critique comme `auth-prod` depend d'un argument de demarrage pour restaurer son contrat (`--import-realm`), versionner aussi cet argument dans la config runtime du container et le verifier en preflight; le `CMD` image seul ne suffit pas.
- Quand `auth-prod` reimporte un realm Keycloak, verifier tout de suite si le realm live contient encore au moins un `super_admin`; un realm `ready` avec `0` users explique des "identifiants invalides" meme si l'OIDC semble sain.
- Quand un wrapper shell repo lance un binaire outil sous `.meta/.tools`, calculer la racine depuis le vrai niveau du script (`../..` ici) et verifier le chemin effectif avant de conclure que l'outil manque.
- Quand Sonar demande `String.raw` sur un matcher ou une regex Next, utiliser `String.raw` directement au lieu de doubler les backslashes dans une chaine classique; c'est plus lisible et evite un faux positif recurrent.
- Quand Pylance remonte `alembic.op` dans une migration, preferer un import explicite du sous-module `alembic.op` et centraliser les littéraux DDL repetes en constantes locales plutot que de disperser la meme chaine partout.
- Quand une migration Alembic expose `op` de facon dynamique mais que Pylance insiste encore, tolerer un ignore Pylance cible sur l'import plutot qu'une refactorisation qui complique le runtime ou le style du module.
- Quand Ruff signale une ligne trop longue dans une migration Alembic, reflowe la colonne ou la chaine en multi-ligne plutot que de compenser avec une constante artificielle si la logique reste plus lisible ainsi.
- Quand Sonar repere des literals SQLA/Alembic repetes comme `now()` ou une FK souvent reutilisee, les centraliser en constantes locales dans la migration pour garder la revision compacte et eviter des doublons triviaux.
- Quand une migration admin back-office repete plusieurs FKs et index temporels, extraire aussi le literal de tri (`created_at DESC`) en constante locale pour eviter les index composites verbeux et garder le scan de revue plus simple.
- Quand une migration operatio​​​​nnelle repete plusieurs FKs et timestamps, regrouper aussi les références de tables et le `now()` en constantes locales pour conserver une lecture par tables et non par litterals.
- Quand une migration de conversation repete seulement les timestamps `created_at`/`updated_at`, introduire un `NOW_SQL` local suffit, sans sur-extraire des constantes de table si le gain n'est pas utile.
- Quand une migration RGPD repete `now()` et `created_at DESC`, centraliser ces literals en constantes locales garde le code lisible sans alourdir la logique append-only.
- Quand un downgrade Alembic re-copie la meme logique de policies plusieurs fois, extraire d'abord un helper de restauration partage pour garder le corps de `downgrade()` sous la limite de complexite sans changer la politique RLS.
- Quand une migration BPM repete `now()` et des JSONB vides `'"'"'{}'"'"'::jsonb`, introduire des constantes locales pour les defaults de colonne plutot que de laisser les literals se disperser dans chaque `sa.Column`.
- Quand le Keycloak local tourne en `dev-file`, forcer aussi `KC_CACHE=local` dans `infra/docker-compose.yml`; sinon un reboot peut partir en pseudo-cluster `jdbc-ping`, retarder la discovery OIDC et recreer des `fetch failed` cote admin.
- Quand une app Next applique une CSP a nonce, garder les ecrans App Router proteges en rendu dynamique; un prerender `○` casse l'hydratation car les scripts et flux inline ne peuvent pas recevoir un nonce scope par requete.
- Quand un flux client depend de `execute-actions-email`, verifier que le realm versionne et le realm live enregistrent vraiment la required action `UPDATE_PASSWORD`; avoir `requiredActions=["UPDATE_PASSWORD"]` sur l'utilisateur ne suffit pas si le provider n'existe pas dans le realm.
- Quand la gouvernance cible dit que la CI GitHub est l'autorite finale, verifier tout de suite que le README racine, les workflows et le chemin de release racontent exactement la meme chose; une doctrine contradictoire vaut presque autant qu'une absence de garde-fou.
- Quand l'utilisateur corrige le contenu exact qu'il a soumis a un jury ou a un tiers, re-evaluer immediatement sur l'artefact reel soumis et non sur le document le plus proche ou le plus complet present dans le repo.
- Quand un hook monorepo lance des commandes Next.js, restaurer explicitement les `next-env.d.ts` generes avant de sortir; un gate vert qui laisse ces fichiers modifies peut bloquer le `push` sans rapport avec le vrai code applicatif.
- Quand une commande racine monorepo orchestre `build`, `lint`, `typecheck` ou `test`, ne jamais enumerer a la main les workspaces courants; deriver le scope du catalogue `app-*` / `packages/*` et imposer les scripts critiques dans le meme changement.
- Quand un workflow GitHub est cense etre l'autorite de release, ne pas laisser de `workflow_dispatch.inputs` influencer `ref`, `services`, `tag` ou la promotion; utiliser le SHA du run et des environnements proteges a la place.
- Quand un contrat runtime machine-readable derive des groupes `all_of` / `any_of`, ne jamais les aplatir en listes `required_*_keys`; preserv­er la semantique de groupe et la tester explicitement, sinon la plateforme ment sur ses vrais prerequis.
- Quand une exception securite passe de `active` a `resolved` ou `revoked`, le validateur ne doit plus la traiter comme expirante/bloquante; garder l'historique gouverne sans obliger a supprimer l'entree du registre.
- Quand un job GitHub utilise `cache: pnpm` avec `actions/setup-node`, installer `pnpm/action-setup` avant dans le meme job; sinon le runner peut echouer tres tot sur `Unable to locate executable file: pnpm`.
- Quand une app consomme un package workspace exporte depuis `dist/`, reconstruire ce package explicitement dans les jobs CI cibles avant `test` ou `build`; un runner propre n'heritera jamais du `dist/` local.
- Quand un composant React exporte un type partage reutilise par un sous-module qu'il importe ensuite, sortir ce type vers un fichier de types dedie ou deja existant; sinon `depcruise` signale un cycle reel meme si TypeScript compile encore.
- Quand j'ajoute une dependance a un workspace, ne pas me fier a la presence globale du package dans `pnpm-lock.yaml`; verifier l'importer exact du workspace ou regenerer le lockfile avant de pousser.
- Quand un workflow CI de surface utilise un guardrail baseline-driven, le scoper au meme perimetre que le workflow; sinon une PR CI-only revele de la dette legacy sans rapport avec la surface verifiee.
- Quand un script bootstrap CI telecharge un outil depuis GitHub Releases, verifier le nom exact de l'asset publie; certains projets livrent un binaire brut et non une archive `.tar.gz`.
- Quand une gate shell CI appelle des validateurs Python repo-owned, ne pas supposer que `python3` du runner embarque `PyYAML`; lancer la gate via `uv run --with pyyaml` ou fournir explicitement la dependance.
- Quand un utilisateur corrige un hero sur le contraste ou le message, relire d'abord le fond reel de la section et la promesse produit canonique avant de pousser une direction visuelle; un bon layout avec une mauvaise base de contraste ou un wording produit faux reste un contre-sens.
- Avant de deployer une branche non-main sur la landing prod, verifier la route publique canonique `https://www.praedixa.com/fr` jusqu'au HTML final et pas seulement le premier `301`; une boucle de redirection auto-referencee peut laisser le container `ready` tout en cassant completement le site.
